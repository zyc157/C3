## 1. 模块概述

### 1.1 模块名称

`[C3级联语音识别纠错翻译模块]`

### 1.2 模块说明

说明本模块在项目中的作用、解决的问题、主要输入输出，以及它为什么是完整系统中必要的一部分。这里可以根据实际实现自由展开。

```text
[本模块属于端到端语音翻译整体系统里的级联子模块，整体流程为英文语音输入→ASR 转录→LLM 自纠错→LLM 翻译。
核心解决原始 ASR 语音识别结果存在错词、漏词、同音错误，错误直接传递影响最终翻译质量的问题；不依赖标准答案真值，依靠大模型语言能力完成纯自纠错，输出干净英文文本和最终中文译文。
输入：原始英文音频文件
输出：原始 ASR 转录文本、LLM 修正英文文本、中文翻译文本、WER 评估指标结果
价值：优化级联语音翻译上游文本质量，降低词错误率，提升整体翻译准确度，可离线独立运行、也可接入整体语音翻译系统]
```

### 5.3 进阶功能 2：`[LLM 无真值自纠错 + WER 指标量化评估]`

#### 功能说明

说明该进阶功能具体做了什么、解决了基础版本的什么不足、对团队完整系统有什么帮助。

```text
[基础版本存在级联系统固有的误差累积问题：Whisper ASR识别产生错字、漏词、发音混淆错误后，错误文本会直接传入翻译环节，导致译文同步失真；且基础版本仅能输出翻译结果，无法量化识别错误程度。
本进阶新增独立LLM文本纠错模块，**不依赖数据集标准参考文本**，仅依靠大模型语言能力修复ASR识别缺陷；同时集成jiwer工具计算纠错前、后两组WER词错误率，用数值直观量化纠错效果。
解决基础版本无识别修复、无量化评估的短板；优化级联流水线上游输入质量，降低误差传递，输出带纠错文本与评估指标的完整数据，可供团队系统做语音识别鲁棒性分析。]

## 6. 与团队系统的集成说明

说明个人模块如何被团队完整系统调用。

建议写清楚：

- 6.1 团队系统调用入口

  团队整体语音调度主程序位于项目根目录 `assignment_C/main_run.py`，通过**子进程调用**方式调用本 C3 级联模块，调用路径：

  `assignment_C/C3_cascade/code/c3_cascade.py`（基线）、`c3_cascade_ex.py`（Qwen3 拓展）、`c3_cascade_ec.py`（纠错评估版）。

​	团队系统统一调度流程：C1 音频预处理 → C3 级联翻译 → C4 端到端对比 → C5 TTS 语音合	成，C3 作为中间核心文本翻译模块，输出结果供 C4、C5 读取。

- 6.2 调用传入参数与文件

1. 必传输入文件
   - 团队统一数据集：`common_data/dataset.json`，全模块共用一套标注，保证跨模块样本对齐；
   - 音频资源：`common_data/audio/` 下全部 wav 音频，由 C1 预处理后标准化为 16kHz 单声道音频输入 C3。

- 6.3 本模块对外返回结果

模块执行完成后生成 2 类 JSON 文件，团队系统读取文件完成跨模块数据互通：

1. ```
   cascade_results.json
   ```

   （基线，ex/ec 对应独立后缀文件）

   

   每条样本包含统一

   ```
   id
   ```

   主键，字段：音频路径、标准英文参考文本、ASR 中间英文文本、中文翻译译文；

   

   C4 端到端模块读取该文件，实现同样本逐条翻译结果对照；C5 TTS 读取

   ```
   translation_zh
   ```

   字段，完成语音合成。

2. ```
   c3_summary.json
   ```

   

   存储总样本数、各阶段推理耗时、模型名称、WER 指标（ec 版本），供团队总脚本汇总全模块性能报表。

- 6.4 依赖资源说明（联调必须提前部署）

1. **数据集**：`common_data/` 完整文件夹，C1/C2/C3/C4/C5 共用，不可缺失；

2. 模型权重

   - 基线：`whisper-small`、`Qwen2.5-1.5B-Instruct`

   - 拓展 ex：额外本地缓存

     ```
     Qwen3-1.7B-Instruct
     ```

     

     团队环境统一提前下载至本地，运行时加

     ```
     --offline
     ```

     避免下载超时；

3. **环境依赖**：统一`speech` conda 虚拟环境，预装 torch、transformers、jiwer、librosa 等库，所有模块共用一套环境；

4. **中间文件**：无临时中间缓存文件，所有结果持久化至`outputs`目录，不占用内存缓存。

   

5. 6.5 联调接口不一致问题与解决方案

​	问题 1：样本 ID 字段命名不统一，跨模块无法匹配样本

​	现象：C1 输出样本标识为`sample_id`，C3 脚本原始字段为`id`，C4 读取时无法对齐对比翻译结果；

  	解决：统一全模块主键字段为`id`，同步修改数据集 json、C1 预处理脚本、C3/C4 读取逻辑，所有输出结果统一使用`id`作为唯一关联键。

问题 2：音频路径相对路径解析错乱，跨目录调用找不到 wav 文件

- 现象：团队主脚本在项目根目录运行，C3 内部相对路径读取音频失败，出现文件不存在报错；
- 解决：C3 脚本增加绝对路径解析逻辑，通过数据集文件位置自动拼接音频完整绝对路径，不受执行目录影响，同时对外输出音频绝对路径写入结果 json。

问题 3：模型下载网络冲突，多模块同时拉取 HuggingFace 模型卡死

- 现象：团队并行运行 C3、C4 时，多个脚本同时访问 HuggingFace 镜像，下载速度慢、进程阻塞；
- 解决：统一配置全局`HF_ENDPOINT=https://hf-mirror.com`镜像，提供离线运行参数`--offline`，团队环境提前一次性预下载全部模型至本地，联调时全部使用离线模式。

问题 4：输出 JSON 格式不标准，C5 读取翻译文本报错

- 现象：C3 早期输出译文存在换行、多余空格，TTS 读取文本出现合成杂音 / 截断；
- 解决：在 LLM 推理后增加文本清洗逻辑，去除换行、多余空格、特殊符号，输出单行干净中文译文，同时统一 JSON 缩进、编码为`utf-8`并开启`ensure_ascii=False`，保证中文正常解析

## 7.已知问题与后续改进

| 问题                                                         | 当前原因                                                     | 后续改进                                                     |
| ------------------------------------------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 1. LLM 翻译串行推理，批量吞吐低、整体运行速度慢              | 代码内 ASR 采用 batch 批量处理，但翻译环节单条循环调用 Qwen 模型，未实现批量输入；大模型逐句推理 GPU 利用率不足 | 实现 LLM 批量翻译逻辑，将多条 ASR 文本拼接批量送入模型推理，提升 GPU 利用率；增加多线程异步推理加速 |
| 2. 纠错模块仅依靠文本语义修复，无法修复音频噪声带来的底层识别偏差 | 纠错逻辑仅处理 ASR 输出文本，没有结合音频声学特征；噪声、语速异常导致的底层识别错误无法从文本层面完全修正 | 对接 C1 音频增强模块，将降噪、扰动后的干净音频送入 ASR；在纠错前增加音频置信度过滤，低置信度样本优先重识别 |
| 3. 缺少多语种级联能力，仅支持英→中单向翻译                   | 提示词、模型调用逻辑固定为英文转中文，未做多语言分支适配     | 增加语种参数`--src_lang`/`--tgt_lang`，适配多语言语音输入；拓展提示词模板，支持中英、中日等双向翻译 |
| 4. 无流式实时推理，仅支持完整音频离线批量处理                | Whisper 与 Qwen 均采用完整音频一次性推理，未接入流式分块识别、增量生成接口 | 接入 Whisper 流式 ASR，搭配 Qwen 增量流式生成，实现近实时语音翻译，适配在线对话场景 |
| 5. 显存占用高，低配置 GPU 运行容易 OOM                       | 未开启模型量化、梯度检查点等显存优化手段，默认加载 FP16 完整权重 | 新增 4bit/8bit 量化加载参数；支持模型分片加载、梯度检查点，降低显存占用，适配低配实训显卡 |
